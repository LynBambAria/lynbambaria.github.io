三、全国大学生建模大赛

LRFMC模型和K-Means聚类

什么是RFM模型，什么是LRFMC模型

这两个模型都是非常经典的用户价值分析模型，广泛用于客户关系管理（CRM）和精细化运营中。简单来说，RFM 是基础版，而 LRFMC 是针对特定行业（如航空、大额消费）的进化版。

  • L（Length，会员入会时长/成为会员的时间长度）： 用户加入会员的时间。入会越久，通常粘性越高，但也可能是“沉睡老客户”。

  • R（Recency，最近一次乘坐/消费时间间隔）： 同上，最近一次乘机距今多久。

  • F（Frequency，消费频率）： 特定时间段内的乘机次数。

  • M（Monetary，消费金额）： 特定时间段内的总花费（或舱位等级对应的价值）。

  • C（舱位等级/距离系数）： 这是一个新增的维度，用于衡量用户是否倾向于乘坐高等级舱位（头等舱/商务舱）或长途飞行。

为什么选择改进RFM模型为LRFMC？

  • 增加 L（入会时长）： 在航空业，一个老会员即使最近一年飞得少，但其历史贡献和品牌忠诚度依然存在，不能简单将其归为“流失客户”。L 维度可以区分“新晋高价值用户”和“资深老客户”。

  • 增加 C（舱位等级）： 同样是消费 5000 元，可能是买了一次经济舱，也可能是买了一次商务舱。利润率完全不同。C 维度能更好地识别真正的高利润贡献者。

将 5 个维度进行打分和聚类，可以将用户分为几类：

  • 高价值/忠诚用户： L 长，R 近，F 高，M 高，C 高（通常是商务人士）。

  • 潜力发展用户： L 较短，R 近，F 中等，M 中等（新注册但活跃的用户）。

  • 一般挽留用户： L 长，R 远（很久没飞了），F 曾经很高（沉睡的老会员）。

  • 低价值用户： 各项指标都低（价格敏感型，偶尔买一次特价票）。

什么是K-Means聚类

K-Means 聚类算法是一种非常经典且应用广泛的无监督学习算法，主要用于将数据划分为若干个组（即“簇”），使得同一组内的数据点相似度尽可能高，而不同组间的数据点相似度尽可能低。

K-Means聚类核心算法

K-Means 通过迭代的方式不断优化，主要包含 4 个步骤：

  1. 初始化： 随机选择 K 个点作为初始的“簇中心”。

  2. 分配： 对于数据集中的每一个点，计算它到 K 个中心的距离，把它归给距离最近的那个中心（比如，离红心近的归为红组，离蓝心近的归为蓝组）。

  3. 更新： 所有点分配完毕后，每个簇内部重新计算“中心点”（通常是计算该簇所有点的平均值位置），将旧的簇中心移动到新的均值位置。

  4. 迭代： 重复步骤 2 和 3，直到满足以下任一条件停止：

  • 簇中心的位置几乎不再变化（收敛）。

  • 达到了预设的最大迭代次数。

你是怎么确定K值的？有没有用过肘部法则或轮廓系数？

业务解释性验证

  • 最终K值需要结合业务可解释性
  • 例如：K=5对应"重要价值客户、重要保持客户、新客户、流失客户、低价值客户"

肘部法则 (Elbow Method)

  • 计算不同K值下的SSE(误差平方和)
  • 寻找"拐点"——K增大时SSE下降速度明显减缓的点

四、网络与分布式系统创新大赛

MOG2

什么是MOG2算法

MOG2是一种自适应的背景建模算法，通过为每个像素维护多个高斯分布来处理复杂的动态背景，并能自动检测阴影。它是传统计算机视觉中做运动检测（如安防监控、人流统计、交通流量检测）的基石算法之一。

MOG2算法核心

  1. 初始化：读取前几帧，为每个像素初始化几个高斯分布。

  2. 像素匹配：对于新的帧，将当前像素的值与它对应的已有的高斯分布进行匹配。

  3. 模型更新：

  • 如果匹配上了某个分布，就用新的像素值更新那个分布的参数（均值和方差）。

  • 如果没有匹配上，且分布数量未达上限，则创建一个新的分布；如果已达上限，则替换掉最不可能代表背景的那个分布。

  1. 前景/背景分割：

  • 将所有的高斯分布按照优先级（权重/标准差）排序。

  • 选取前B个分布作为背景模型（B由算法根据权重和自适应阈值决定）。

  • 如果当前像素匹配上了这前B个分布中的任何一个，则该像素是背景（显示为黑色）；否则，是前景（显示为白色）或阴影（显示为灰色）。

形态学操作

腐蚀

就像“侵蚀”一样，它会收缩或细化图像中的前景物体（通常白色部分）。

效果：

  • 消除物体边界的毛刺和不规则凸起。

  • 分离两个原本有细微连接的物体。

  • 去除小的、孤立的噪声点（因为这些点太小，结构元素无法完全放进去）。

膨胀

与腐蚀相反，它就像“扩张”一样，会增长或加粗图像中的前景物体。

效果：

  • 填充物体内部的细小空洞。

  • 连接两个原本距离很近的物体。

  • 使物体的边界向外扩张，加粗物体。

开运算

先腐蚀，后膨胀。

效果：

  • 主要用于去除图像中细小的噪声点，并平滑较大物体的边界，同时不会明显改变物体的面积。

闭运算

先膨胀，后腐蚀。

效果：

  • 主要用于填充物体内部的细小空洞，并连接邻近的物体，同时不会明显改变物体的面积。

五、智能健康检测手表

请简要介绍一下这个项目

这是一个基于 ESP32-S3 的独立全栈项目，我负责从硬件选型、嵌入式固件开发到 Flutter 移动端 APP 的完整链路。最大的挑战在于资源受限条件下的多目标权衡——在 512KB SRAM 的约束下，同时满足实时心率算法、30fps 图形界面、稳定 BLE 连接和低功耗待机的需求。

比如内存分配上，我将 LVGL 双缓冲区分配合计控制在 150KB 以内，为心率算法预留 80KB 缓冲区，BLE 协议栈占用约 100KB，剩余留给 FreeRTOS 内核和任务栈。当发现帧率不稳定时，我通过 lv_timer_handler() 的调用频率调整和任务优先级重新分配，而非简单增加缓冲区。

ESP32-S3 选型理由

维度 ESP32-S3 优势 对比说明
AI 指令集 内置 Vector 指令集 (SIMD)，支持加速 DSP 运算 虽然心率算法没用上，但为后续 PPG 信号 ML 处理预留空间
内存配置 512KB SRAM + 384KB ROM，支持外部 PSRAM 最高 8MB STM32F4 系列通常 128-256KB，需外接 RAM 才能跑 LVGL
无线性能 BLE 5.0 + Wi-Fi 双模，射频性能稳定 对比 nRF52 系列需外接屏幕驱动，S3 单芯片解决
图形加速 支持 2D 图形加速 (DMA + 专用指令) LVGL 在 S3 上可利用 esp_lcd 驱动实现硬件加速
功耗控制 支持 ULP 协处理器 + 多种低功耗模式 比 ESP32 功耗降低 30%，且支持更精细的电源域控制
生态工具 ESP-IDF 成熟，支持 JTAG 调试、功耗分析工具 STM32 生态分散，ESP 统一工具链降低开发成本

FreeRTOS 任务划分与调度

四个核心任务：

  1. 传感器采集任务 (vTaskSensorAccel)
  • 频率：100Hz（加速度计），25Hz（心率传感器）
  • 机制：使用 二值信号量 由硬件定时器触发，确保采样间隔精确
  • 数据：写入 循环缓冲区 (Ring Buffer)，供算法任务消费
  1. 心率算法任务 (vTaskHeartRateAlgo)
  • 触发：当缓冲区积累 5 秒数据（125 个 PPG 样本）时启动
  • 处理：带通滤波（0.5-4Hz）+ 峰值检测 + 运动伪影消除
  • 输出：通过 队列 发送心率值到 UI 任务和 BLE 任务
  1. UI 刷新任务 (vTaskUIRefresh)
  • 机制：调用 lv_timer_handler()，配合 vTaskDelayUntil() 实现固定 33ms 周期（30fps）
  • 优化：使用 双缓冲区 + DMA 传输，CPU 仅在渲染时活跃
  • 通信：通过 事件组 (Event Group) 接收心率更新、蓝牙状态变化等事件
  1. 蓝牙通信任务 (vTaskBLEComm)
  • 模式：基于 ESP-IDF 的 NimBLE 协议栈
  • 机制：使用 队列 接收待发送数据，主循环阻塞在 ble_gap_event() 回调
  • 流控：实现 滑动窗口 + ACK 确认 机制，防止数据堆积

我刻意将心率算法设为独立高优先级任务，而非在传感器中断中处理，是因为 PPG 信号处理涉及浮点运算和滤波器，耗时约 50-80ms。如果在中断中执行会导致加速度计采样抖动，独立任务配合信号量触发既能保证实时性，又不阻塞其他任务。

心率传感器和加速度计分别用的是什么型号？你通过什么接口（I2C/SPI）读取数据？在读取时遇到过什么问题（如总线死锁、数据错乱）？

MAX30102，ADXL345。都是I2C。调试时发现 MAX30102 偶尔无响应，查看日志发现 I2C 死锁：从机拉低 SDA 未释放。根本原因是 ESP32-S3 在深度休眠唤醒时，I2C 控制器复位但传感器仍处于发送状态。

协议层面：启用 I2C 超时检测，避免无限等待

你是如何实现“心率监测响应时间 < 2秒”的？

  1. 快速启动优化
    MAX30102 从休眠到稳定采样需 500ms，我采用 预加热机制：检测到抬腕动作（加速度计中断）提前 300ms 启动心率传感器

  2. FreeRTOS 实时保障

  • 心率算法任务优先级设为 3，高于 UI 和 BLE
  • 使用 信号量 由传感器数据就绪中断触发，避免轮询开销
  • 算法执行时间 80ms，远小于采样周期 40ms（25Hz），留有充足余量

实测从手指接触传感器到显示心率，平均 1.4 秒，最坏情况（低灌注度）1.8 秒。这 2 秒包含了 500ms 硬件启动 + 800ms 数据积累 + 100ms 算法处理
